# Provx — Project Anchor

**This is the canonical entry point for any AI assistant working in this workspace.** Read this file before doing anything non-trivial.

Provx is a governed, deterministic, auditable automated security-validation platform (web, API, AD/infra over time) for small pentest firms and consultants and for internal security/compliance teams — including air-gapped / no-AI environments. It fills the gap between autonomous "AI hacker" tools (non-reproducible, LLM-required, unsafe for production) and expensive commercial suites: Provx runs a safe-by-default engagement workflow (scope/RoE enforcement, passive vs active modes, per-finding human-approved exploitation) that produces reproducible, compliance-grade results and client-ready reports. AI is an **optional advisor, never required**. Open-core, Apache-2.0, DCO-signed, single monorepo. Positioning is deliberate: Provx is **not** an autonomous AI hacker — see [docs/POSITIONING_and_STRATEGY.md](docs/POSITIONING_and_STRATEGY.md).

## Operating mode (the senior baseline)

For ANY work in this project, operate as a senior software architect / senior full-stack / senior UI-UX / senior DBA / senior security engineer with experience equivalent to a top-tier company (Google, Apple). Apply:

- **SOLID + DRY** (without over-engineering)
- **Reuse before create** — search the codebase for existing helpers/components before writing new ones (rule Q-11)
- **No over-comments** — code should be self-explanatory; comments explain WHY, not WHAT (rule Q-12)
- **ISO 27001 / OWASP Top 10** awareness
- **User-friendly error messages** in production — never leak server / SQL / third-party / stack-trace errors (rule S-13, B-FA-06, **PX-ERRORS**). Detailed errors only when `APP_ENV` is `local` / `development` / `testing` / `staging` (gate via env var).
- **Tests run with `--env=testing` or `.env.testing`** (rule Q-10)
- **Read context BEFORE writing code** — workflow rule, see `feedback_seniority_and_workflow.md` in memory

These behavioral defaults are auto-loaded into every session via the project memory.

## Project state (as of 2026-07-19)

Pre-alpha, building toward the **v0.1 MVP** (Web module only). The **walking skeleton works end to end**: `POST /engagements` → scope-gated scan via the SDK's `security_headers` adapter (discovered as an entry-point plugin) → normalized Findings sealed with SHA-256 → PostgreSQL via SQLModel + Alembic → findings list and HTML report, with a Next.js Server Component page over the top.

Also live: the scoped HTTP egress boundary (`provx_sdk.fetch.fetch_within_scope`, per-hop scope re-check — see PX-EGRESS), the scan-time safety gate (`SAFE_MODE` + engagement `mode` + `adapter.safety`), and a lab with a vulnerable and a clean target scoring TP/FP/FN as a real CI gate. CI runs ruff, mypy --strict, pytest, tsc, and the accuracy gate.

**Not built yet:** authentication/RBAC, the arq job queue (scans run inline), the deterministic playbook engine (expressions are stored verbatim and structure-validated only until the restricted evaluator exists — see PX-DSL), Active mode, exploitation, additional adapters, AI (off by default and optional forever), pgvector, hosted SaaS. The Roadmap gantt starts ~2026-08; MVP target ~2026-09.

Known-but-deferred defects live in [docs/KNOWN_ISSUES.md](docs/KNOWN_ISSUES.md) — check there before reporting a bug as new.

## What NOT to do

- ❌ **Don't turn Provx into an autonomous AI hacker.** AI is an optional advisor only; no code path may hard-depend on an LLM (PX-AI-OPTIONAL). Keep it fully usable with AI off.
- ❌ **Don't use `eval` / `exec` / `compile`+exec / `pickle` in the playbook/expression engine** — playbook `when`/`if` are untrusted input (PX-DSL). Until the restricted evaluator exists, store expressions verbatim and structure-validate only.
- ❌ **Don't build MVP-out-of-scope features** without an explicit decision: API module, AD/Infra, exploitation runners, cloud/K8s, mobile, scheduling, third-party integrations, attack-path graphs (ROADMAP §4). AI Autopilot is demoted to an optional module, not a headline.
- ❌ **Don't auto-exploit or run intrusive/active work without gating + recorded authorization** (PX-ACTIVE, PX-EXPLOIT). Exploitation needs per-finding human approval, sandboxing, non-destructive proof, and a replay trail.
- ❌ **Don't skip scope enforcement, and don't open a second egress path.** All outbound HTTP goes through `provx_sdk.fetch.fetch_within_scope`, which re-checks scope on every redirect hop. Never construct an `httpx`/`requests` client elsewhere, and never set `follow_redirects=True` (PX-SCOPE, PX-EGRESS).
- ❌ **Don't log secrets/credentials/tokens/sessions**, and don't mutate the audit log — it's append-only (PX-SECRETS, PX-EVIDENCE). Hash + timestamp evidence at capture.
- ❌ **Don't absorb GPL/AGPL tool source** (sqlmap, nmap, OpenVAS) into the codebase — invoke them as subprocesses only (PX-LICENSE).
- ❌ **Don't leak internals in client-facing errors** — sanitize; expose detail only when `APP_ENV` is dev/local (PX-ERRORS, S-13, B-FA-06).
- ❌ **Don't modify `docs/`** casually — it's the canonical decision record; changes there are deliberate governance decisions.
- ❌ **Don't commit AI attribution** in commit messages or leave AI-tell characters/prose in output (rules Q-13, Q-15).
